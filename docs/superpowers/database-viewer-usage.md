# Просмотрщик баз данных — как пользоваться

Встроенный в этот форк Zed инструмент для быстрого просмотра баз данных PostgreSQL:
дерево коннекшенов, структура таблиц, просмотр строк с сортировкой/фильтрами/пагинацией,
SQL-вкладка и MCP-сервер для AI-агентов.

## Открыть панель

Иконка базы данных в левом доке открывает панель «Databases». Если её не видно,
вызовите действие `database_panel::ToggleFocus` из палитры команд (Cmd+Shift+P →
«database panel»).

## Добавить подключение

1. Кнопка `+` в шапке панели → форма подключения.
2. Заполните: имя (уникальное), host, port (по умолчанию 5432), база, пользователь, пароль.
3. «Test Connection» проверяет доступность; «Save» сохраняет.

Параметры подключения пишутся в `settings.json` (секция `database.connections`),
**пароль хранится только в macOS Keychain** и на диск не попадает.

Пример секции настроек:

```jsonc
"database": {
  "page_size": 100,            // строк на страницу
  "query_timeout_seconds": 30, // таймаут запроса
  "mcp_max_rows": 200,         // лимит строк в ответе MCP-инструмента
  "connections": [
    { "name": "local-shop", "host": "127.0.0.1", "port": 5432, "database": "shop", "user": "postgres" }
  ]
}
```

Коннекшены можно править и руками в `settings.json`; пароль после этого задаётся
через «Edit Connection» в контекстном меню (правый клик по коннекшену).

## Просмотр данных

- Клик по коннекшену раскрывает базы → схемы → таблицы (загрузка ленивая).
- Клик по таблице открывает вкладку с двумя режимами:
  - **Data**: SQL-first вид — под шапкой виден и редактируется реальный SQL,
    который выполняется, ниже строкой чипы активных фильтров/сортировки, затем
    сетка с данными, внизу — футер с пагинацией и правкой (подробности ниже).
  - **Structure**: колонки (тип, nullable, default, PK, FK) и индексы.

### SQL-бар

Под шапкой вкладки Data всегда виден сгенерированный запрос:

- Запрос можно править прямо в этом редакторе и запускать **Cmd+Enter**
  (macOS) / **Ctrl+Enter** (Windows/Linux) или кнопкой **Run**.
- Шеврон слева сворачивает бар до одной строки «SQL» (когда мешает) и
  разворачивает обратно.
- Как только текст запроса отредактирован и запущен, вкладка переходит в
  **custom-режим**: под бар добавляется бейдж «Custom query · read-only» —
  сортировка/фильтры из UI и правка ячеек в этом режиме недоступны (сама
  таблица не может быть однозначно определена для UPDATE/DELETE). Клик по
  заголовку колонки в custom-режиме сортирует, оборачивая исходный запрос в
  подзапрос, а не переписывая его.
- Кнопка **«Reset to table query»** рядом с бейджем возвращает к обычному
  сгенерированному запросу по таблице (сортировки/фильтры/пагинация из UI).
- Отдельно от custom-режима таблица может быть read-only по другой причине
  (вьюха, нет первичного ключа) — тогда в футере показывается краткая причина,
  например «Read-only: view» или «Read-only: no primary key».

### Фильтры

Есть два способа добавить фильтр:

- **Правый клик по ячейке** — контекстное меню предлагает «Filter: колонка =
  'значение'» и «Exclude: колонка ≠ 'значение'» (для NULL — «Filter: колонка IS
  NULL» / «Exclude: колонка IS NOT NULL»).
- **Воронка (funnel-иконка)** в заголовке колонки (появляется при наведении,
  остаётся видимой, если фильтр уже есть) открывает попап с выбором оператора
  (`=`, `≠`, `>`, `<`, `contains`, `is null`, `is not null`) и полем значения.

Активные фильтры и сортировка отображаются чипами в строке под SQL-баром:
клик по чипу переоткрывает попап для правки, крестик на чипе снимает фильтр
(или сортировку) и перезапускает запрос.

### Сортировка

Клик по заголовку колонки циклически меняет сортировку: не сортировано →
по возрастанию → по убыванию → снова не сортировано. В custom-режиме клик по
заголовку не переписывает введённый SQL, а оборачивает его в подзапрос с
`ORDER BY` снаружи.

### Пагинация

Внизу сетки — стрелки «Previous page» / «Next page», счётчик диапазона строк
(«rows 1–100», «No rows» или «{N} rows» для custom-запроса без `LIMIT`) и
рядом переключатель размера страницы (`100 / page`, `500 / page`, `1000 / page`
плюс значение из настроек, если оно отличается от этих трёх).

## SQL-запросы

Правый клик по коннекшену или базе → «New SQL Query» открывает вкладку с редактором
SQL и таблицей результатов. Запуск — **Cmd+Enter** (macOS) / **Ctrl+Enter** (Linux).
Сессии UI открыты в режиме read-write, поэтому из SQL-вкладки можно выполнять и
`UPDATE`/`INSERT`/`DELETE`, а не только `SELECT`.

## Редактирование данных

Во вкладке **Data** базовые таблицы с первичным ключом можно редактировать прямо в сетке.
Вьюхи, таблицы без первичного ключа и результаты custom-запроса (см. SQL-бар выше)
доступны только для чтения — причина показывается в футере.

- **Правка ячейки**: двойной клик по ячейке открывает инлайн-редактор; Enter — принять
  изменение в буфер, Esc — отменить. Изменённые ячейки подсвечиваются янтарным.
  Кнопка **∅ NULL** в редакторе задаёт ячейке значение NULL (отличается от пустой строки).
  Колонки первичного ключа у существующих строк не редактируются (они идентифицируют
  строку; чтобы сменить ключ — удалите строку и добавьте новую).
- **Добавить строку**: кнопка **+ Row** в футере добавляет пустую строку (зелёная). Её
  ячейки редактируются так же; незаполненные колонки при вставке берут значения по
  умолчанию (строка целиком без заполнения вставляется как `DEFAULT VALUES`).
- **Удалить строку**: наведите на строку и нажмите иконку корзины. Существующая строка
  помечается на удаление (зачёркнута, красная), новая — просто убирается.
- **Сохранить/Отменить**: когда есть изменения, в футере (рядом с переключателем размера
  страницы) появляется «N change(s)» с кнопками **Save** и **Discard**. Save применяет
  все правки **одной транзакцией** (UPDATE/INSERT/DELETE) на read-write сессии; при
  ошибке (нарушение NOT NULL, типа, внешнего ключа, уникальности, или если строка
  изменилась параллельно) вся транзакция откатывается, ошибка показывается целиком, а
  буфер сохраняется — можно исправить и повторить. Discard отменяет все накопленные
  правки.
- **Обновить**: справа в футере — время последнего запроса («N rows · M ms») и кнопка
  **Refresh** (перезапускает текущий запрос).

Ограничения: редактирование только для базовых таблиц с первичным ключом в обычном
(не custom) режиме; правка значений PK на месте не поддерживается; оптимистичная
блокировка — по первичному ключу (каждая UPDATE/DELETE должна затрагивать ровно одну
строку).

## Доступ для AI-агентов (MCP)

Отдельный бинарь `zed-database-mcp` даёт агентам доступ к тем же базам **только на
чтение** (сессии MCP жёстко read-only, что бы агент ни прислал). Он использует те же
коннекшены из `settings.json` и пароли из Keychain.

Инструменты: `list_connections`, `list_tables`, `describe_table`, `run_query`.

### Подключить к агенту Zed

Соберите бинарь и пропишите его как context server:

```bash
cargo build --release -p database_mcp   # → target/release/zed-database-mcp
```

```jsonc
// settings.json
"context_servers": {
  "zed-database": { "command": "/path/to/zed/target/release/zed-database-mcp", "args": [] }
}
```

### Подключить к Claude Code

```bash
claude mcp add zed-database /path/to/zed/target/release/zed-database-mcp
```

## Что пока не поддерживается (планы)

- Другие СУБД (MySQL/MariaDB, SQLite) — архитектура готова к добавлению.
- SSH-туннели (пока — вручную `ssh -L`), обязательный TLS/sslmode.
- Экспорт результатов и сохранение `.sql`-файлов.
