# Docker-панель — как пользоваться

Встроенная в этот форк Zed панель для управления Docker: список подключений
(локальный демон + SSH-хосты), дерево контейнеров/образов/compose-проектов на
каждом из них, базовые действия (старт/стоп/рестарт, pull/remove образа,
compose up/down/restart), просмотр логов и `docker inspect`.

## Открыть панель

Иконка сервера в левом доке открывает панель «Docker». Если её не видно,
вызовите действие `docker_panel::ToggleFocus` из палитры команд (Cmd+Shift+P →
«docker panel»).

## Подключения (endpoints)

По умолчанию панель всегда показывает синтетический эндпоинт `local`
(обращение к локальному Docker-демону через стандартный сокет, без
`DOCKER_HOST`). Дополнительные эндпоинты — локальные alias'ы или удалённые
хосты по SSH — добавляются в `settings.json`, секция `docker.connections`:

```jsonc
"docker": {
  "poll_interval_seconds": 300, // интервал автообновления статуса, сек; 0 — выключить автообновление
  "connections": [
    { "name": "prod", "kind": "ssh", "ssh_host": "deploy@1.2.3.4", "read_only": true },
    { "name": "staging", "kind": "ssh", "ssh_host": "deploy@stg" }
  ]
}
```

Поля записи подключения:

- **`name`** — уникальное отображаемое имя эндпоинта.
- **`kind`** — `"local"` (демон на этой же машине, сокет по умолчанию) или
  `"ssh"` (удалённый демон, доступ через `DOCKER_HOST=ssh://...`).
- **`ssh_host`** — обязателен при `kind: "ssh"`, вид `user@host`.
- **`read_only`** — булево, по умолчанию `false`. См. ниже.

Никаких паролей или ключей в настройках нет: аутентификация SSH-подключения
целиком идёт через ssh-agent пользователя и `~/.ssh/config` — то же самое,
что происходит при обычном `ssh user@host`. Панель не хранит и не запрашивает
секреты.

### Добавление/редактирование/удаление эндпоинтов из панели

Эндпоинты необязательно прописывать в `settings.json` руками — панель это
умеет сама:

- **Добавить**: кнопка «+» в шапке панели (рядом с кнопкой обновления)
  открывает диалог «Add Endpoint». Поля: **Name** (уникальное имя), **Kind**
  (переключатель Local/SSH, по умолчанию SSH), **SSH host** (`user@host`,
  показывается и обязателен только при Kind = SSH), **Read-only** (чекбокс,
  по умолчанию выключен — раз пользователь сам добавляет эндпоинт, скорее
  всего он собирается им управлять). Кнопка **Test Connection** проверяет
  досягаемость демона (`docker version` через выбранный Kind/SSH host) прямо
  из диалога, не сохраняя ничего — это не блокирует Save, а лишь подсказка.
  **Save** записывает запись в `docker.connections` в `settings.json` и
  закрывает диалог; **Cancel** просто закрывает его.
- **Редактировать/удалить**: правый клик по строке эндпоинта в дереве
  открывает контекстное меню. Пункты «Edit Endpoint…» и «Remove Endpoint»
  показываются, только если эндпоинт настроен через `docker.connections`
  (т.е. это не синтетический `local` и не авто-импортированный контекст —
  см. ниже): у обоих нет соответствующей записи в настройках, поэтому
  редактировать/удалять для них нечего. При редактировании поле Name
  заблокировано (имя — идентификатор записи в настройках), остальные поля
  предзаполнены текущими значениями.
- Изменения применяются сразу: панель хранит своё состояние в
  `DockerEndpointStore`, который сам переподписан на изменения настроек и
  пересинхронизируется при любом сохранении `settings.json` — вручную
  обновлять список эндпоинтов после Save/Remove не нужно.
- Как и раньше, диалог не имеет поля пароля и ничего не пишет в keychain:
  аутентификация SSH целиком на стороне ssh-конфига пользователя.

### Авто-импорт контекстов

При старте (и при каждом обновлении настроек) панель дополнительно запускает
`docker context ls` на локальной машине и добавляет обнаруженные контексты как
эндпоинты, если у них нет одноимённого совпадения среди настроенных
(включая синтетический `local`). Настроенный в `settings.json` эндпоинт
всегда побеждает при конфликте имён и сохраняет свой собственный `read_only`.
Для импортированных так эндпоинтов `read_only` зависит от их вида:

- **удалённые (`ssh`) контексты** импортируются как `read_only: true`
  (безопасно по умолчанию: авто-обнаруженный удалённый хост может оказаться
  продакшеном) — чтобы управлять таким эндпоинтом (выполнять деструктивные
  действия), пропишите его явно в `docker.connections` с тем же именем и
  `read_only: false`;
- **локальные контексты** (сокет по умолчанию, `default` и подобные)
  импортируются как `read_only: false` — это демон на машине самого
  пользователя, им можно управлять сразу, без ручной настройки.

Если `docker context ls` недоступен или завершился ошибкой, это не ломает
панель: список эндпоинтов остаётся как есть (настроенные + `local`).

## Флаг `read_only`

`read_only: true` блокирует на этом эндпоинте **любое** изменяющее действие —
не только деструктивные (`stop`/`restart` контейнера, `remove` образа,
`compose down`/`compose restart`/`compose up`), но и недеструктивные (`start`
контейнера, `pull` образа тоже блокируются). Read-only эндпоинт полностью
view-only: разрешены только чтения (`list`, `logs`, `inspect`), которые вообще
не проходят через этот шлюз. Проверка выполняется на уровне обработчика
действия (`DockerEndpointStore::dispatch_action` — единая точка, через
которую проходит каждое действие), а не только видимостью кнопки в UI,
поэтому даже сбой в интерфейсе не приведёт к вызову `docker` для read-only
эндпоинта.

Деление на деструктивные/недеструктивные действия при этом никуда не делось —
оно определяет только необходимость подтверждения (см. ниже) на **writable**
эндпоинте: `start` контейнера и `pull` образа выполняются сразу же, без
модального окна, а вот `stop`/`restart`/`remove`/`compose down`/`compose
restart`/`compose up` требуют подтверждения.

## Подтверждение деструктивных действий

Перед любым деструктивным действием (stop/restart/remove/compose down и
подобные) панель показывает модальное окно подтверждения с точной командой,
которая будет выполнена (например, `docker compose -p shop down`), и именем
эндпоинта. Действие выполняется только по нажатию «Confirm»; «Cancel» и
закрытие окна ничего не запускают.

## Дерево эндпоинта

Эндпоинт, у которого `read_only: true`, помечен в дереве маленькой иконкой
замка рядом с именем (и подсказкой «read-only endpoint» при наведении) — так
видно, какие эндпоинты чисто для просмотра, ещё до раскрытия дерева.

Клик по эндпоинту раскрывает три раздела: **Containers**, **Images**,
**Compose**. Выбор строки в дереве открывает деталку справа:

- **Container**: статус, порты; кнопки Start/Stop/Restart, состояние которых
  зависит от текущего статуса контейнера и от `read_only` эндпоинта, а не
  только от типа действия:
  - **Start** активна, только если контейнер НЕ запущен и эндпоинт writable;
    иначе — подсказка «container is already running» (если запущен) или
    «endpoint is read-only» (если read-only).
  - **Stop**/**Restart** активны, только если контейнер запущен (включая
    `Restarting`/`Paused`) и эндпоинт writable; иначе — подсказка «container
    is not running» или «endpoint is read-only».
  - Подтверждение по-прежнему требуется для Stop/Restart на writable
    эндпоинте (Start остаётся без подтверждения).
  Logs и Inspect всегда активны (это чтения, `read_only` их не касается) и
  открываются как полноразмерные вкладки в центральной рабочей области (как
  SQL/таблицы у database-панели), а не внутри узкого дока: Logs — потоковый
  хвост `docker logs -f` с переключателем follow/pause, Inspect — полный
  вывод `docker inspect` в виде JSON. Закрытие вкладки Logs останавливает
  поток и завершает дочерний процесс `docker logs -f`.
  Вкладка Logs рендерит строки плотно, «по-терминальному» (моноширинный
  шрифт буфера, компактный line-height — без больших зазоров между
  строками), поддерживает ANSI-цвет (частые SGR-коды — reset/bold/обычные
  и яркие 30-37/90-97 цвета текста — раскрашиваются, остальные escape-
  последовательности просто вырезаются, а не показываются как мусор) и
  имеет поле поиска/фильтра в тулбаре: по мере ввода запроса видимые строки
  сужаются до тех, что регистронезависимо содержат подстроку, рядом
  показывается счётчик совпадений («N / M matches»); сам буфер логов при
  этом продолжает накапливаться полностью, фильтруется только то, что
  показано.
- **Image**: репозиторий/тег/размер; Pull (без подтверждения) и Remove (с
  подтверждением).
- **Compose project**: статус; Up/Down/Restart (Up и Restart — с
  подтверждением, кроме случая когда `read_only` уже заблокировал бы вызов).

## Автообновление

Панель периодически обновляет статус каждого не-ошибочного эндпоинта:
интервал задаётся `docker.poll_interval_seconds` (по умолчанию 300 секунд, т.е.
5 минут) и применяется на следующем тике без перезапуска. Для SSH-эндпоинтов
реальный интервал не может быть меньше 15 секунд, даже если в настройках
указано меньшее значение — это защищает удалённый демон от слишком частых
опросов.

Значение `"poll_interval_seconds": 0` полностью отключает автообновление
(только ручное обновление). В этом режиме обновлять статус нужно кнопкой
Refresh в шапке панели (иконка в виде кругового значка) — она вызывает
действие `docker_panel::RefreshEndpoint`, которое обновляет выбранный в
дереве эндпоинт, а если ничего не выбрано — все эндпоинты сразу.

## Что входит в v1

Список действий, поддержанных в этой версии панели:

- Контейнеры: list, start, stop, restart, inspect, logs.
- Образы: list, pull, remove.
- Compose: up, down, restart (list проектов).

## Чего пока нет (Phase 2)

MCP-инструменты для AI-агентов (по аналогии с `zed-database-mcp`: read-инструменты
list/logs/inspect и деструктивные действия за тем же флагом `read_only` плюс
шагом propose/confirm) в этой версии **не реализованы**. Панель — чисто
интерактивный UI-инструмент для человека.
