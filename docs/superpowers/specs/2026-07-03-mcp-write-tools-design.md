# MCP: запись в базу с двухфазным подтверждением — дизайн (этап 4)

Дата: 2026-07-03
Статус: дизайн одобрен пользователем
Базируется на: `2026-07-02-database-viewer-design.md` (MCP-сервер `zed-database-mcp`,
строго read-only), `2026-07-03-data-editing-design.md` (выделенные read-write
соединения, транзакционный apply)

## Мотивация

Пользователь хочет, чтобы агент (Zed agent / Claude Code) мог по явной просьбе
записывать данные в базу: «запиши в такую-то БД такие-то данные». При этом
мутация обязана быть предварительно показана — что было до и что станет после —
и применяться только после подтверждения. Агент не должен иметь возможность
закоммитить то, чего не показывал.

## Цель

Два новых MCP-инструмента с серверным двухфазным протоколом
propose → preview → apply. Запись включается явно, per-connection, по умолчанию
выключена. Существующий `run_query` остаётся строго read-only без изменений.

## Не-цели

- DDL (CREATE/ALTER/DROP/TRUNCATE) — запрещён. Только DML: INSERT / UPDATE / DELETE.
- Лимит на число затрагиваемых строк — НЕ вводится (решение пользователя);
  превью показывает точный охват, решение за подтверждающим.
- Изменения UI-части (database_ui) — не требуются.
- Многостейтментные скрипты, COPY, вызовы функций с побочными эффектами через
  SELECT — вне охвата (propose_write принимает один DML-стейтмент).

## Инструменты

### `propose_write`

Вход: `{ "connection": string, "database": string, "sql": string }`.

1. **Гейт**: коннекшен должен иметь `"allow_mcp_writes": true` в настройках;
   иначе isError с текстом «writes are disabled for this connection; set
   allow_mcp_writes in settings».
2. **Классификация**: после срезания ведущих пробелов и SQL-комментариев
   (`--` и `/* */`) первое ключевое слово (регистронезависимо) должно быть
   `INSERT`, `UPDATE` или `DELETE`; ровно один стейтмент — `;` допустима только
   как завершающая. Всё остальное (SELECT, WITH, DDL, множественные
   стейтменты) — isError с объяснением. SELECT-ы агент выполняет через
   существующий `run_query`.
3. **Превью** (выделенное соединение `SessionMode::ReadWrite` через существующий
   `connect_dedicated`; кэшированная read-only сессия MCP не используется):
   - `BEGIN`;
   - выполнить стейтмент, дописав `RETURNING *`, если в тексте нет своего
     `RETURNING` (если есть — использовать пользовательский);
   - собрать `rows_affected` и возвращённые строки;
   - `ROLLBACK` — в базе ничего не изменилось;
   - для UPDATE дополнительно «до»: после ROLLBACK выполнить
     `SELECT * FROM <таблица> WHERE (pk...) IN (значения PK из RETURNING)` —
     PK берётся из метаданных таблицы (тот же запрос, что у describe_table).
     Если у таблицы нет PK, UPDATE меняет сами PK-колонки или таблицу не
     удалось однозначно определить из стейтмента — «до» опускается, в ответе
     пометка `"before": null, "note": "..."`.
   - Семантика превью: INSERT → `after` = вставляемые строки, `before` = null;
     DELETE → `before` = удаляемые строки (RETURNING отдаёт их до удаления),
     `after` = null; UPDATE → `after` из RETURNING + `before` по PK.
   - Превью строк капится существующим `mcp_max_rows` (настройка, дефолт 200);
     при усечении — пометка `"preview_truncated": true` (rows_affected всегда точный).
4. **Токен**: случайный одноразовый id (16+ байт из системного ГПСЧ),
   хранится в памяти процесса MCP в таблице
   `token -> { connection, database, sql, created_at }`; TTL 5 минут;
   одноразовый (удаляется при apply или по истечении). Токен привязан к
   ТОЧНОМУ тексту sql — apply_write не принимает sql на вход вовсе, только токен.

Ответ (JSON в text-контенте, как у остальных инструментов):
`{ "token", "statement_kind": "insert|update|delete", "rows_affected",
"columns", "before": [...]|null, "after": [...]|null,
"preview_truncated": bool, "expires_in_seconds", "note"? }`.

### `apply_write`

Вход: `{ "token": string }`.

1. Найти и удалить proposal по токену (несуществующий/истёкший → isError
   «unknown or expired token; call propose_write again»).
2. Повторно проверить гейт `allow_mcp_writes` (настройки могли измениться).
3. На новом выделенном read-write соединении: `BEGIN` → выполнить ТОТ ЖЕ sql
   (с тем же RETURNING) → `COMMIT`. Ошибка на любом шаге → ROLLBACK + isError.
4. Ответ: `{ "rows_affected", "columns", "returned": [...],
   "rows_affected_matches_preview": bool }` — если реальное число строк
   разошлось с превью (данные изменились между propose и apply), это видно.

Гонка propose↔apply признана допустимой: apply выполняет тот же SQL по
текущему состоянию базы; расхождение отражается в ответе, откатывать его не
нужно (пользователь подтверждал стейтмент, охват показан повторно).

## Настройки

В `settings_content` `DatabaseConnection` добавляется поле
`allow_mcp_writes: Option<bool>` (дефолт false; отсутствие = false), плюс
дефолт в `assets/settings/default.json` (в шаблоне connections его нет — поле
опциональное, документируется в usage-доке). MCP-бинарь уже читает
settings.json напрямую — донести поле до его конфига коннекшена.

## Слои защиты (сводка)

1. Per-connection флаг, по умолчанию запись выключена везде.
2. Двухфазность: превью считается сервером в откатываемой транзакции; токен
   одноразовый и привязан к точному SQL — применить неотображённое нельзя.
3. Подтверждение вызова apply_write в MCP-клиенте (Zed/Claude Code) остаётся
   обычным механизмом апрува тулов на стороне пользователя.
4. Только DML, один стейтмент; run_query остаётся под BEGIN READ ONLY.

## Размещение кода

Транзакционная логика превью/apply живёт в `database_client` (новые методы
трейта `DatabaseClient`, реализация в `postgres.rs` на выделенном соединении —
рядом с `apply_edits`, зеркало в `fake.rs`); классификатор стейтментов,
токен-стор и сами инструменты — в `database_mcp` (`tools.rs`). UI-крейт не
затрагивается.

## Обработка ошибок

- SQL-ошибка на превью (синтаксис, нарушение констрейнта) → ROLLBACK,
  isError с текстом Postgres — агент видит проблему до какого-либо коммита.
- Обрыв соединения между propose и apply → apply вернёт ошибку соединения,
  токен уже потрачен → повторный propose.
- Несколько соседних propose живут независимо (разные токены).
- Процесс MCP перезапустился → токены пропали → apply честно отказывает.

## Тестирование

- Юнит: классификатор стейтментов (INSERT/UPDATE/DELETE/отказы: SELECT, WITH,
  DDL, `;` в середине, комментарии перед ключевым словом); дописывание
  RETURNING; извлечение имени таблицы для UPDATE-«до» (простые формы
  `UPDATE [ONLY] schema.table ...`); токен-стор (TTL, одноразовость).
- Интеграционные с FakeDatabaseClient — если трейт расширяется, зеркалить в
  фейке; протокольные тесты propose→apply happy-path и отказов (стиль
  существующих тестов database_mcp).
- Живые тесты против Docker-Postgres (`shop`): UPDATE с before/after по PK,
  DELETE preview, INSERT preview, ROLLBACK-гарантия (после propose данные не
  изменились), apply коммитит, повторный apply тем же токеном отказывает,
  таблица без PK → before=null с note.
