# Переработка страницы таблицы: SQL-first UX — дизайн (этап 3)

Дата: 2026-07-03
Статус: дизайн одобрен пользователем (макет + секции)
Базируется на: `2026-07-02-database-viewer-design.md` (этап 1),
`2026-07-03-data-editing-design.md` (этап 2)

## Мотивация

Пользователь попробовал текущий просмотрщик и ощущает «связанные руки»:

1. Нет редактируемого SQL на странице таблицы — в JDBC-клиенте VS Code при
   открытии таблицы сразу показывается сгенерированный запрос
   (`SELECT * FROM table LIMIT 100`), который можно править на месте.
2. Конструктор фильтров (кнопка «+ Filter» → три дропдауна + Apply) неинтуитивен.
3. Сортировка неочевидна, пагинация неудобна (нет выбора размера страницы).
4. Таблица данных выглядит грубо (нет зебры, NULL неотличим от пустой строки,
   ширины колонок не подстраиваются, нет ховера).

## Цель

Сделать страницу данных таблицы SQL-first: видимый и редактируемый запрос —
источник истины, все UI-действия переписывают его текст. Фильтрация — через
ячейки и заголовки. Пагинация — страницы с выбором размера. Общая полировка
внешнего вида грида.

## Не-цели

- Панель дерева коннекшенов — без изменений.
- SQL-вкладка (`SqlQueryView`) — без изменений; остаётся для многострочных скриптов.
- MCP-сервер — без изменений (read-only модель не трогаем).
- Механика сохранения правок (`apply_edits`, транзакции, PK-гейт) — без изменений.
- Экспорт/копирование результатов, multi-column sort, подсчёт точного COUNT —
  вне этого этапа.

## Компоновка страницы (сверху вниз)

1. **Шапка вкладки** (существующая): имя таблицы, коннекшен/схема, переключатель
   Data · Structure.
2. **SQL-бар**: редактор Zed с подсветкой SQL (как в SQL-вкладке), авто-высота
   1–5 строк, сворачивается шевроном слева. Справа кнопка **Run** (⌘⏎/Ctrl+Enter).
   В custom-режиме — жёлтый бейдж «Custom query · read-only» и кнопка
   «Reset to table query».
3. **Строка чипов**: активные фильтры и сортировка. Чип фильтра:
   `status = 'active' ×`; чип сортировки: `total desc ×`. Крестик снимает,
   клик по чипу фильтра открывает попап редактирования. Строка скрыта, когда
   чипов нет.
4. **Грид данных** (изменённый рендер, см. «Полировка вида»).
5. **Футер**: ◀ ▶, счётчик `rows 1–100+`, выпадающий выбор размера страницы
   (варианты 100 / 500 / 1000; дефолт — `database.page_size` из настроек, и если
   он не из списка — добавляется в список отдельным пунктом), кнопка
   «+ Row», Save/Discard c «N changes» при непустом буфере правок, справа —
   `N rows · M ms` и иконка Refresh (повторный Run — на случай свёрнутого
   SQL-бара).

Тулбар этапов 1–2 (Refresh, «+ Filter», старые кнопки) поглощается этой
компоновкой; отдельного тулбара больше нет.

## Модель поведения: QueryState

Единый инвариант: **текст в SQL-баре всегда в точности равен запросу, который
выполняется**.

```
QueryState {
    base: Base,                  // Table(TableRef) | Custom(String)
    filters: Vec<Filter>,        // как сейчас + новый FilterOp::IsNotNull
    sort: Option<(column, dir)>,
    limit: usize,                // размер страницы
    offset: usize,
}
```

`render(&QueryState) -> String` — чистая детерминированная функция:

- `Base::Table`: плоский запрос
  `SELECT * FROM "schema"."table" [WHERE …] [ORDER BY …] LIMIT n OFFSET m;`
  Значения фильтров — строковые литералы с экранированием (удвоение `'`);
  Postgres приводит типы неявно (заменяет текущие касты
  `$N::text::"udt_schema"."udt_name"` в UI-пути).
- `Base::Custom`: если фильтры/сортировка/пагинация активны — обёртка
  `SELECT * FROM (<текст без завершающей ;>) AS zed_sub [WHERE …] [ORDER BY …]
  LIMIT n OFFSET m;`; если надстройки нет — текст как есть.

### Переходы режимов

- Открытие таблицы → `Base::Table`, limit = `database.page_size`, offset 0;
  текст = `render(state)`; запрос сразу выполняется.
- UI-действие (сортировка кликом, фильтр, страница, размер) → мутирует
  `QueryState`, текст перегенерируется `render`-ом, запрос перезапускается.
- Ручная правка текста: как только текст ≠ `render(state)` («dirty»),
  редактирование ячеек блокируется. По Run dirty-текст фиксируется:
  `base = Custom(текст)`, фильтры/сортировка сбрасываются, offset = 0 —
  страница в custom-режиме.
- UI-действие в custom-режиме мутирует надстройку → текст перегенерируется
  (видна обёртка `zed_sub`) и перезапускается.
- Повторная ручная правка обёрнутого текста + Run → новый `Custom(текст)`,
  надстройка снова сбрасывается.
- «Reset to table query» → `base = Table(исходная таблица)`, надстройка
  сбрасывается, offset 0, перезапуск.

### Гейт редактируемости

Редактирование ячеек/строк доступно только при `base == Table` **и** прежних
условиях этапа 2 (базовая таблица с PK). В custom-режиме грид read-only:
инлайн-редактор, «+ Row», Delete Row и контекст-пункт «Edit cell» отключены.
Все операции, меняющие страницу (Run, сортировка, фильтры, пагинация, Reset),
сначала вызывают `finish_editing` — инвариант этапа 2 сохраняется.

## Фильтры

- **Правый клик по ячейке** → контекстное меню:
  - `Filter: col = 'значение'` (для NULL-ячейки — `Filter: col IS NULL`),
  - `Exclude: col ≠ 'значение'` (для NULL — `col IS NOT NULL`),
  - `View value` — попап с полным значением (моноширинно, с переносами),
  - `Edit cell` (только когда редактируемо).
- **Воронка в заголовке колонки** (видна при наведении; постоянно и цветом —
  когда по колонке есть фильтр) → компактный попап: дропдаун оператора
  (`=`, `≠`, `>`, `<`, `contains`, `is null`, `is not null`) + поле значения +
  Apply. Тот же попап открывается кликом по чипу (предзаполненный).
- Новый оператор `FilterOp::IsNotNull` в `database_client`.
- Несколько фильтров соединяются AND (как сейчас). Старый конструктор
  «+ Filter» удаляется полностью.

## Сортировка и пагинация

- Клик по заголовку циклит: asc → desc → нет; в заголовке стрелка ▲/▼.
  Одна колонка сортировки (как сейчас).
- Футер: ◀ ▶ переписывают `offset`; выбор размера страницы переписывает
  `limit` (и сбрасывает offset). Всё через `render` → текст → перезапуск.
- `has_more`-эвристика: LIMIT выполняется ровно как в тексте (без скрытого
  `n+1`); если строк вернулось == limit — счётчик показывает `1–100+` и ▶
  активна. Пустая следующая страница — нормальное состояние (◀ доступна).
  Для custom-запроса без надстройки счётчик — просто `N rows`.

## Полировка вида

- Зебра (чередование фона строк) + подсветка строки под курсором.
- Шапка: отчётливый фон, разделитель снизу, имена колонок medium-weight.
- NULL — серым курсивом `NULL`; пустая строка — действительно пустая ячейка.
- Значения — моноширинным шрифтом (buffer font); числовые колонки (int2/4/8,
  numeric, float4/8) выравниваются вправо, когда тип известен из структуры
  (только `Base::Table`; в custom — всё влево).
- Автоширина колонок по содержимому первой страницы: ширина ≈ максимальная
  длина значения/заголовка, кламп [60px, 480px]; ручной ресайз
  (`ResizableColumnsState`) работает поверх и сохраняется, пока вкладка открыта.
- Длинные значения обрезаются с «…»; полное — через «View value».

## Техническая сторона

- Новый модуль `crates/database_ui/src/query_state.rs`: `QueryState`, `render`,
  экранирование литералов, обёртка custom-запроса, снятие завершающей `;`.
  Чистые юнит-тесты без GPUI.
- Путь данных страницы таблицы переводится с `fetch_rows(SelectSpec)` на
  `run_query(текст)` (та же UI read-write сессия, тот же cancel/timeout).
  Семантика выполнения текста идентична SQL-вкладке. `fetch_rows` и его
  bind-machinery удаляются из `DatabaseClient`/`sql.rs`/`postgres.rs`/`fake.rs`,
  если после перевода не остаётся потребителей (MCP их не использует).
- Ограничение строк UI (существующий кап 1000 в SQL-вкладке) применяется и к
  результату страницы таблицы.
- Сопоставление колонок результата со структурой таблицы (для PK/RowKey,
  выравнивания чисел, редактирования) — по именам колонок; структура уже
  грузится жадно (этап 2).
- `table_data_view.rs` (3776 строк) разгружается: `QueryState` — отдельный
  модуль; рендер грида и попапов остаётся, старый конструктор фильтров и
  тулбар удаляются.

## Обработка ошибок

- Ошибка запроса (синтаксис в custom, недоступность БД) — как сейчас в
  SQL-вкладке: сообщение об ошибке вместо грида, состояние QueryState не
  теряется.
- Обёртка custom-запроса может быть невалидной для не-SELECT текста
  (например, обёрнутый `UPDATE`) — ошибку показывает Postgres; UI ничего не
  парсит и не валидирует заранее.
- Save правок при custom-режиме невозможен по построению (буфер пуст —
  редактирование блокировано до входа в режим).

## Тестирование

- Юнит (`query_state.rs`): render для всех комбинаций base/надстройки,
  экранирование `'`, обёртка custom, снятие `;`, IsNotNull.
- GPUI (fake-клиент): клик сортировки перегенерирует текст и перезапускает;
  ручная правка → dirty → редактирование ячеек заблокировано; Run dirty-текста
  → custom-режим (бейдж, read-only), сброс фильтров; Reset возвращает
  table-режим; фильтр из контекст-меню ячейки добавляет чип и WHERE;
  крестик чипа убирает; пагинация переписывает OFFSET; смена размера — LIMIT
  и сброс offset; `finish_editing` вызывается на всех перезапусках;
  has_more-эвристика.
- Живой прогон против Docker-Postgres (`shop`): фильтры по text/int/numeric
  литералами, custom-запрос с JOIN + сортировка обёрткой, пагинация.
