# Редактирование данных в таблице — дизайн (этап 2)

Дата: 2026-07-03
Статус: дизайн (пользователь пре-одобрил реализацию после плана)
Базируется на: `2026-07-02-database-viewer-design.md` (просмотрщик PostgreSQL)

## Цель

Дать возможность прямо во вкладке данных таблицы редактировать значения ячеек,
добавлять новые строки и удалять строки, с явным сохранением одним нажатием.
Удобно и практично, как редактируемая таблица (grid) в GUI-клиентах БД.

## Требования

### Функциональные

1. **Правка ячейки**: клик выделяет ячейку, двойной клик (или Enter) — инлайн-редактор;
   ввод значения, Enter — принять в буфер изменений, Esc — отменить правку ячейки.
2. **NULL**: в редакторе ячейки отдельная кнопка/шорткат «Set NULL», чтобы отличать
   NULL от пустой строки. Текущее NULL-значение отображается как «NULL» (muted).
3. **Добавление строки**: кнопка «+ Row» добавляет пустую строку внизу; её ячейки
   редактируются; незаполненные колонки при вставке опускаются (берётся DEFAULT).
4. **Удаление строки**: выделить строку → «Delete Row» помечает её на удаление
   (визуально зачёркнута), до сохранения физически не удаляется.
5. **Буфер изменений**: правки/вставки/удаления копятся, подсвечиваются
   (изменённая ячейка — янтарный фон, новая строка — зелёный, удаляемая — красный
   зачёркнутый). Тулбар показывает «N changes» с кнопками **Save** и **Discard**.
6. **Сохранение**: генерирует параметризованные UPDATE/INSERT/DELETE и выполняет их
   **в одной транзакции** на read-write сессии UI; при ошибке — откат, ошибка в UI,
   буфер сохраняется. При успехе — буфер очищается, текущая страница перезагружается.

### Нефункциональные / безопасность

- Значения передаются как текст и кастуются к типу колонки на сервере
  (`$N::text::"udt_schema"."udt_name"`) — тот же приём, что в фильтрах.
- UPDATE/DELETE всегда ограничены полным первичным ключом; каждая операция должна
  затрагивать ровно 1 строку — иначе сообщить (конкурентное изменение/отсутствие строки).
- Всё в транзакции `BEGIN; ...; COMMIT;` (откат при любой ошибке).
- Редактирование доступно **только** для базовых таблиц с первичным ключом.
  Вьюхи и таблицы без PK — только чтение, с явным сообщением почему.
- Колонки первичного ключа не редактируются (они идентифицируют строку); смена PK =
  удалить + вставить. Это осознанное ограничение v1.

### Вне объёма v1

- Массовое редактирование, копирование/вставка диапазонов.
- Оптимистичная блокировка по всем полям (v1 — только по PK, с проверкой rows-affected).
- Редактирование значений PK на месте, каскадные правки.
- Отмена/повтор (undo/redo) отдельных правок сверх Discard-всё.

## Архитектура

### `database_client` (слой БД)

Новый типизированный API построения и применения изменений — в `sql.rs` (построение,
юнит-тесты) и трейте `DatabaseClient` (применение в транзакции).

```rust
// Ячейка редактирования: значение или явный NULL.
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum EditCell { Value(String), Null }

// Идентификатор строки — значения PK-колонок в порядке структуры.
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct RowKey { pub columns: Vec<String>, pub values: Vec<Option<String>> }

pub struct RowUpdate { pub key: RowKey, pub set: Vec<(String, EditCell)> }     // set: колонка → новое значение
pub struct RowInsert { pub values: Vec<(String, EditCell)> }                    // только заданные колонки
pub struct RowDelete { pub key: RowKey }

#[derive(Default)]
pub struct TableEdits {
    pub updates: Vec<RowUpdate>,
    pub inserts: Vec<RowInsert>,
    pub deletes: Vec<RowDelete>,
}

#[derive(Debug, Default, PartialEq, Eq)]
pub struct AppliedCounts { pub updated: usize, pub inserted: usize, pub deleted: usize }
```

- `sql.rs`: pure-функции `build_update`, `build_insert`, `build_delete`, возвращающие
  `(sql, params)` с идентификаторами через `quote_ident` и значениями через
  `$N::text::"udt_schema"."udt_name"` (тип берётся из `ColumnInfo`). WHERE строится по
  `RowKey` (каждая PK-колонка через параметр с кастом). Юнит-тесты на все три + на
  NULL-ячейки (для NULL — литерал `NULL`, без параметра) и на экранирование.
- Трейт `DatabaseClient`: `async fn apply_edits(&self, table: &TableRef, columns: &[ColumnInfo], edits: &TableEdits) -> Result<AppliedCounts>`. Реализация Postgres:
  открыть транзакцию, применить DELETE → UPDATE → INSERT (порядок), для каждой
  UPDATE/DELETE проверить `rows_affected == 1` (иначе `bail!` с понятным текстом →
  откат), COMMIT; при любой ошибке ROLLBACK и вернуть ошибку. `columns` нужны для
  кастов типов и определения, какие из них PK. Fake-клиент журналирует вызов и
  возвращает канированные counts / ошибку.
- Только `SessionMode::ReadWrite`. Для read-only сессий `apply_edits` — ошибка
  (агенты не пишут; MCP этот метод не вызывает).

### `database_ui` (вкладка таблицы)

Расширяем `table_data_view.rs` (не новый файл — редактирование неотделимо от вкладки).

- Гейт редактируемости: при загрузке структуры вычислить `editable: bool` =
  `!is_view && !pk_columns.is_empty()`. Структуру грузить вместе с первой страницей
  данных (сейчас лениво — добавить упреждающую загрузку PK/структуры), чтобы сразу
  знать, включать ли UI-редактирования.
- Состояние правок в `TableDataView`:
  ```rust
  edits: TableEditBuffer, // updates: HashMap<RowKey, HashMap<col, EditCell>>,
                          // inserts: Vec<HashMap<col, EditCell>>,
                          // deletes: HashSet<RowKey>
  editing_cell: Option<EditingCell>, // какая ячейка сейчас в инлайн-редакторе + Entity<InputField>
  ```
- Рендер ячеек: изменённые/новые/удаляемые — с цветовой подсветкой; инлайн-редактор
  на активной ячейке; кнопка «+ Row» и «Delete Row», тулбар «N changes / Save / Discard».
  PK-ячейки не входят в режим редактирования (визуально помечены как ключ).
- Save: собрать `TableEdits` из буфера (RowKey из оригинальных PK-значений строки),
  вызвать `client.apply_edits(...)` через `Tokio::spawn_result`; при успехе очистить
  буфер и `reload_data`; при ошибке — показать `{:#}`-текст, буфер оставить.
- Публичные методы для тестов: `begin_edit_cell`, `set_cell_value`, `set_cell_null`,
  `add_row`, `delete_row`, `discard_edits`, `save_edits(cx)`, геттеры буфера.

## Поток данных (сохранение)

1. Пользователь правит ячейки / добавляет / удаляет строки → буфер + `cx.notify()`.
2. Save → из буфера строится `TableEdits` (ключи — оригинальные PK строки).
3. `apply_edits` в фоне (read-write сессия): транзакция, проверка rows-affected, COMMIT.
4. Успех → буфер очищен, страница перезагружена; ошибка → откат на сервере, текст в UI.

## Обработка ошибок

- Нет PK / вьюха → UI-редактирование недоступно, показать причину (баннер во вкладке).
- Нарушение ограничений (NOT NULL, тип, FK, unique) → ошибка Postgres целиком (`{:#}`)
  в тулбаре/уведомлении; транзакция откатана, буфер цел (пользователь исправляет).
- UPDATE/DELETE затронул ≠ 1 строку → явное сообщение «row not found or changed
  concurrently», откат.
- Разрыв соединения при сохранении → ошибка, буфер цел, повтор возможен.

## Тестирование

- `database_client`: юнит-тесты `build_update/insert/delete` (идентификаторы,
  `$N::text::"schema"."udt"`, WHERE по PK, NULL-литералы, множественные SET/values);
  тест `apply_edits` порядка/счётчиков на fake; ignored live-тест: реальный
  UPDATE+INSERT+DELETE в транзакции против Docker-базы с проверкой rows-affected и
  отката при нарушении (например, INSERT с нарушением NOT NULL откатывает весь пакет).
- `database_ui`: GPUI-тесты с fake: буфер (правка/вставка/удаление/discard), гейт
  editable для вьюхи и таблицы без PK, save вызывает apply_edits с корректным
  `TableEdits`, ошибка сохранения оставляет буфер.
- Ручная проверка в собранном Zed на Docker-базе (правка ячейки, +Row, Delete, Save,
  проверка данных в psql; откат при ошибке).

## Этапы (план — отдельный документ)

1. `database_client`: типы + `build_update/insert/delete` + юнит-тесты.
2. `database_client`: `apply_edits` в трейте + Postgres-реализация (транзакция,
   rows-affected) + fake + ignored live-тест.
3. `database_ui`: буфер правок + гейт editable + упреждающая загрузка структуры.
4. `database_ui`: инлайн-редактор ячейки + NULL + подсветка + тулбар Save/Discard.
5. `database_ui`: +Row / Delete Row + сборка TableEdits + save/reload + тесты.
6. Сквозная проверка на Docker-базе + clippy + финальное ревью.
