# Просмотрщик баз данных для Zed (форк) — дизайн

Дата: 2026-07-02
Статус: одобрен пользователем (brainstorming-сессия)

## Цель

Встроенный в форк Zed инструмент для быстрого просмотра баз данных PostgreSQL —
аналог расширения Database Client для VS Code, но проще pgAdmin. Плюс MCP-сервер,
дающий тот же доступ к базам AI-агентам (агент Zed, Claude Code и другие MCP-клиенты).

## Требования

### Функциональные (v1)

1. Боковая панель с деревом: коннекшены → базы данных → схемы → таблицы и вьюхи.
2. Добавление коннекшена через UI-форму в панели: имя, host, port, база,
   пользователь, пароль, кнопка «Проверить соединение».
3. Клик по таблице открывает вкладку в области редактора с двумя режимами:
   - **Данные**: первые N строк (по умолчанию 100), сортировка кликом по
     заголовку колонки, фильтры по полям, пагинация вперёд/назад.
     Фильтр по полю — выпадающий выбор оператора (`=`, `≠`, `>`, `<`,
     `содержит`, `IS NULL`) плюс поле значения; активные фильтры
     объединяются через AND.
   - **Структура**: колонки (имя, тип, nullable, default), первичные и внешние
     ключи, индексы.
4. Вкладка SQL-запроса: встроенный редактор Zed с подсветкой SQL сверху,
   таблица результатов снизу, запуск по Cmd+Enter.
5. MCP-сервер (отдельный бинарь, stdio) с инструментами:
   `list_connections`, `list_tables`, `describe_table`, `run_query`.

### Нефункциональные

- UI никогда не блокируется: вся работа с БД — в фоновых задачах.
- Большие таблицы не страшны: сортировка/фильтры/пагинация выполняются на
  сервере БД (`SELECT ... WHERE ... ORDER BY ... LIMIT ... OFFSET ...`),
  данные в память целиком не загружаются.
- Пароли хранятся только в macOS Keychain, на диск не пишутся.
- Режимы доступа (уточнено 2026-07-03): сессии UI — read-write (запись из
  SQL-вкладки разрешена; основа для будущего редактирования в таблице);
  сессии MCP — строго read-only (`default_transaction_read_only = on`).
  `PostgresClient` принимает режим сессии параметром.
- Минимальные правки существующего кода Zed, чтобы форк легко подтягивал
  обновления из upstream.

### Вне объёма v1 (осознанно отложено)

- MySQL/MariaDB, SQLite и другие СУБД (архитектура закладывает возможность).
- SSH-туннели (при необходимости пользователь делает `ssh -L` вручную).
- Обязательный TLS/sslmode для управляемых баз (добавим при необходимости).
- Редактирование данных из табличного UI (правка ячеек, INSERT) — этап 2
  (запись из SQL-вкладки разрешена с 2026-07-03).
- Экспорт результатов (CSV и т.п.).

## Архитектура

Три новых крейта, по конвенциям репозитория (`[lib] path` с осмысленным именем,
без `mod.rs`):

```
crates/database_client   — слой БД, без GPUI-зависимостей
crates/database_ui       — панель, вкладки, форма коннекшена (GPUI)
crates/database_mcp      — бинарь zed-database-mcp (MCP-сервер по stdio)
```

### `database_client`

- Трейт `DatabaseClient`: `list_databases`, `list_schemas`, `list_tables`,
  `table_structure`, `fetch_rows(запрос выборки)`, `run_query(sql)`,
  `test_connection`.
- Реализация `PostgresClient` на `tokio-postgres` (runtime — уже имеющийся в
  Zed `gpui_tokio`; в MCP-бинаре — обычный tokio).
- Типы: `ConnectionConfig`, `TableRef`, `ColumnInfo`, `RowsPage`,
  `SelectSpec { filters, sort, limit, offset }` и построитель SQL по нему.
- Метаданные — из `pg_catalog`/`information_schema`.
- В Postgres одно TCP-подключение привязано к одной базе, поэтому коннекшен
  в конфиге указывает стартовую базу, список баз сервера берётся из
  `pg_database`, а раскрытие другой базы в дереве открывает отдельное
  подключение к ней с теми же учётными данными.
- Правила генерации SQL: имена таблиц/колонок экранируются как идентификаторы,
  значения фильтров передаются только параметрами запроса ($1, $2, ...),
  конкатенация пользовательских значений в SQL запрещена.
- Фейковая реализация трейта для тестов UI.

### `database_ui`

- **Панель** (`Panel`, по образцу `project_panel`): дерево коннекшенов,
  ленивое раскрытие узлов (списки баз/схем/таблиц запрашиваются при
  раскрытии), красная иконка с tooltip при ошибке подключения, кнопка «+».
- **Форма коннекшена**: поля имя/host/port/база/пользователь/пароль,
  «Проверить соединение», сохранение: параметры — в settings.json через
  `update_settings_file`, пароль — в Keychain через существующий механизм Zed
  (`credentials_provider`).
- **Вкладка таблицы** (`Item`, по образцу `csv_preview`): компонент
  `ui::Table` (ресайз колонок), переключатель «Данные / Структура», строка
  фильтров, пагинация, индикатор загрузки.
- **Вкладка SQL** (`Item`): встроенный `Editor` с языком SQL + `ui::Table`
  результатов, действие «Выполнить» (Cmd+Enter), отображение ошибок Postgres
  в области результатов, кнопка отмены запроса.
- Регистрация: `database_ui::init(cx)` и добавление панели в
  `initialize_panels` в `crates/zed/src/zed.rs` (минимальные правки),
  members в корневом `Cargo.toml`.

### `database_mcp`

- Бинарь `zed-database-mcp`, MCP по stdio на официальном Rust SDK (`rmcp`).
- Переиспользует `database_client`; читает коннекшены из settings.json Zed и
  пароли из Keychain — конфигурация общая с панелью, отдельной нет.
  При первом обращении бинаря к Keychain macOS может показать системный
  запрос на доступ — это ожидаемо, разрешается один раз.
- Инструменты: `list_connections`, `list_tables(connection)`,
  `describe_table(connection, table)`, `run_query(connection, sql)`.
- Ограничения: read-only-сессия, таймаут запроса, максимум 200 строк в ответе
  инструмента (настраиваемо), чтобы не раздувать контекст агента.
- Подключение: в форке Zed регистрируется как context server для агента;
  для Claude Code — `claude mcp add`.

## Настройки (пример)

```jsonc
// settings.json
{
  "database": {
    "page_size": 100,            // строк на страницу в UI
    "query_timeout_seconds": 30,
    "mcp_max_rows": 200,
    "connections": [
      {
        "name": "local",
        "host": "127.0.0.1",
        "port": 5432,
        "database": "postgres",
        "user": "postgres"
        // пароль — в Keychain, здесь его нет
      }
    ]
  }
}
```

Форма в панели пишет в этот же файл — конфиг можно править и руками.

## Поток данных

1. Действие в UI (раскрытие узла, клик по заголовку, смена страницы, запуск SQL)
   формирует запрос к `DatabaseClient`.
2. Запрос выполняется фоновой задачей (`gpui_tokio`); в UI — спиннер.
3. Результат возвращается на foreground-поток, entity обновляется, `cx.notify()`.
4. Ошибки не глотаются: доводятся до UI (см. ниже) в соответствии с
   правилами CLAUDE.md репозитория.

## Обработка ошибок

- Ошибка подключения → красная иконка у коннекшена в дереве + уведомление
  workspace с текстом ошибки.
- Ошибка запроса (синтаксис, права) → текст ошибки Postgres в области
  результатов вкладки.
- Таймаут (по умолчанию 30 с, настраиваемо) → сообщение + возможность отмены.
- Отмена запроса → реальный cancel-сигнал серверу
  (`tokio_postgres` cancel token), не только скрытие спиннера.
- Разрыв соединения → пометка коннекшена и автоматический реконнект при
  следующем действии.

## Тестирование

- Юнит-тесты `database_client`: построение SQL из `SelectSpec`
  (экранирование идентификаторов, параметры, LIMIT/OFFSET), разбор метаданных.
- GPUI-тесты `database_ui` с фейковым `DatabaseClient`: раскрытие дерева,
  сортировка, пагинация, отображение ошибок.
- Тесты инструментов MCP на фейковом клиенте.
- Ручная проверка: локальный Postgres в Docker с тестовыми данными;
  проверка панели, вкладок, SQL и подключения MCP к агенту Zed и Claude Code.
- Линт: `./script/clippy`.

## Этапы реализации

1. Тулчейн: установка rustup, первая сборка форка (~15–30 мин; каталог
   сборки до ~30–50 ГБ на диске).
2. `database_client` + юнит-тесты (проверяем подключение к живой базе рано).
3. Панель с деревом + форма коннекшена + Keychain.
4. Вкладка таблицы: данные/структура, сортировка, фильтры, пагинация.
5. Вкладка SQL-запросов.
6. `database_mcp` + регистрация в агенте Zed, проверка из Claude Code.

Работа ведётся в ветке `database-viewer`; `main` форка остаётся синхронизируемым
с upstream.
