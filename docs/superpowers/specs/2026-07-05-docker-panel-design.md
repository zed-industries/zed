# Docker-панель для форка Zed — дизайн

**Статус:** одобрено (брейншторм 2026-07-05).
**Ветка:** `database-viewer` (тот же форк, что и БД-панель).
**Автор процесса:** субагент-исполнение, пользователь делегирует автономную реализацию (Rust не пишет сам).

## 1. Цель

Управлять Docker прямо из редактора — локально и на удалённых серверах по SSH — для повседневных действий: смотреть контейнеры и их статус, читать/стримить логи, перезапускать/останавливать/запускать, работать с образами и docker-compose-стэками. Панель повторяет удачный паттерн БД-панели форка: дерево слева → детали справа, действия с явным подтверждением разрушающих операций.

## 2. Объём

**v1 (этот дизайн):**
- **Контейнеры:** список со статусом, `start` / `stop` / `restart`, `inspect` (порты, mounts, env, created), просмотр и `-f`-стрим логов.
- **Образы:** список (repo:tag, size, id), `pull`, `remove`.
- **Compose:** обнаружение стэков, список сервисов, `up` / `down` / `restart` сервиса/стэка.
- **Эндпоинты:** локальный сокет + удалённые по `ssh://user@host`; авто-импорт `docker context ls` + ручные записи в настройках.
- **Безопасность:** confirm-модалка на разрушающие действия; per-endpoint флаг `read_only` (на нём разрушающие действия заблокированы).

**Не в v1 (позже):** exec/шелл в контейнер; volumes/networks/stats/prune; MCP-инструменты для агента (Фаза 2); Swarm/Kubernetes; сборка образов (`docker build`).

**Не-цели:** заменить `docker` CLI целиком; редактор Dockerfile/compose (это обычные файлы, уже редактируются); оркестрация продакшена.

## 3. Архитектура

Зеркалит структуру БД-фичи (`database_client` / `database_ui` / `database_mcp`).

- **`docker_client`** (новый крейт; `[lib] path = "docker_client.rs"`): абстракция доступа к Docker за трейтом.
  - `trait DockerClient: Send + Sync` — методы `list_containers`, `list_images`, `list_compose_projects`, `inspect_container`, `container_logs` (стрим), `start`/`stop`/`restart_container`, `pull_image`, `remove_image`, `compose_up`/`compose_down`/`compose_restart`.
  - `CliDockerClient` — реальная реализация через шелл-аут к `docker` / `docker compose` с `--format json`, задавая `DOCKER_HOST` под эндпоинт. Стрим логов — построчное чтение stdout дочернего процесса.
  - `FakeDockerClient` — для тестов: канонные списки/ошибки, запись вызванных команд.
  - Типы данных: `DockerEndpoint`, `Container`, `ContainerStatus`, `Image`, `ComposeProject`, `ComposeService`, `LogChunk`, `DockerError`.
- **`docker_ui`** (новый крейт, GPUI): панель в доке, дерево эндпоинтов, детали, действия, вью логов, confirm-модалки, read-only-гейт. Регистрируется как workspace-панель тем же способом, что и БД-панель.
- **`settings_content`**: новый `crates/settings_content/src/docker.rs` с `DockerConnectionContent`, добавляется ключ настроек `docker.connections` (по образцу `database.connections`).
- **Фаза 2 — `docker_mcp`** (отдельный бинарь): read-инструменты (список/логи/inspect) без гейта; разрушающие — за тем же `read_only`-флагом и явным подтверждением (паттерн propose/apply из `database_mcp`).

## 4. Модель эндпоинтов и настройки

Схема `DockerConnectionContent`:
```
{
  "name": "local",                 // отображаемое имя, уникально
  "kind": "local" | "ssh",         // local = дефолтный сокет
  "ssh_host": "user@host",         // требуется при kind=ssh; идёт в DOCKER_HOST=ssh://user@host
  "read_only": false               // true → разрушающие действия заблокированы
}
```
- **Секреты не хранятся.** SSH-аутентификация — из ssh-agent/`~/.ssh/config` пользователя (в отличие от паролей БД, Keychain не нужен). Это осознанное упрощение: панель не управляет SSH-ключами.
- **Авто-импорт контекстов:** при старте панель читает `docker context ls --format json` и предлагает существующие контексты как эндпоинты (дедуп по имени с ручными записями из настроек). Ручные записи в настройках имеют приоритет.
- Локальный эндпоинт по умолчанию присутствует всегда (без `DOCKER_HOST`).

## 5. Транспорт

`docker` CLI + `--format json`, `DOCKER_HOST` под эндпоинт (`ssh://user@host` для remote). Обоснование: покрывает «local + по SSH» без самодельного туннелирования сокета и без TLS-плумбинга; переиспользует SSH-конфиг пользователя. Требует наличие бинаря `docker` локально (у пользователя есть).
- Чтение: `docker ps -a --format json`, `docker images --format json`, `docker inspect <id>`, `docker compose ls --format json` / `docker compose ps`.
- Логи: `docker logs -f --tail N <id>` со стримом stdout.
- Действия: `docker start|stop|restart <id>`, `docker pull <ref>`, `docker rmi <id>`, `docker compose -p <proj> up -d|down|restart [service]`.
- Разбор JSON устойчивый: у части команд вывод — JSON-lines (по объекту на строку); парсим построчно, неизвестные поля игнорируем.

## 6. Панель (UI)

- **Дерево слева:** `Эндпоинт → {Контейнеры, Образы, Compose} → элементы`. У контейнеров статус-точка (running/exited/paused), у эндпоинта — состояние подключения (OK/Connecting/Error).
- **Детали справа** (по выбранному элементу):
  - Контейнер: имя, image, статус, порты, mounts, env, created; кнопки Start/Stop/Restart, Logs, Inspect(raw JSON).
  - Образ: repo:tag, id, size, created; Pull (обновить), Remove.
  - Compose-проект: список сервисов со статусом; Up/Down/Restart для стэка и отдельных сервисов.
- **Вью логов:** стрим `docker logs -f`, прокрутка, тумблер «follow», кнопка стоп/старт стрима.
- **Refresh:** автополлинг статусов раз в ~5 с (настраиваемо) + ручной refresh; поллинг для remote-эндпоинтов не чаще, чтобы не нагружать SSH.

## 7. Действия и безопасность

- Read-действия (список/inspect/логи) доступны всегда.
- Разрушающие (stop/restart/remove/compose down/up):
  - На эндпоинте с `read_only: true` кнопки заблокированы с подсказкой «endpoint is read-only».
  - На управляемом эндпоинте — confirm-модалка, показывающая **точную команду и целевой эндпоинт** («`docker restart api` на `prod`») перед выполнением. Тот же этос явного подтверждения мутации, что и в write-пути БД.
- Действие выполняется, статус элемента обновляется по факту; ошибка показывается в UI.

## 8. Поток данных и конкурентность

- Каждый вызов Docker — дочерний процесс на фоновом executor (`gpui_tokio`), foreground-задача применяет результат к entity панели и `cx.notify()`.
- Стрим логов — длящаяся задача, хранится в поле вью; отменяется при закрытии/смене выбора (drop `Task`).
- Поллинг — таймер через `cx.background_executor().timer(...)` (правило проекта для таймеров), а не `smol::Timer`.

## 9. Обработка ошибок

- Нет бинаря `docker` → эндпоинт в Error с явным сообщением и подсказкой установить Docker.
- SSH-фейл / демон недоступен / контекст не резолвится → эндпоинт Error с текстом stderr (обрезанным).
- Правила проекта: ошибки пробрасываем `?`, при намеренном игноре — `.log_err()`; никаких `let _ =` на fallible-операциях; без `unwrap()/expect()` вне тестов; осторожно с индексацией при парсинге вывода.
- Ошибка одного действия/эндпоинта не роняет панель целиком.

## 10. Тестирование

- **`docker_client`:** `FakeDockerClient` + юнит-тесты на (а) сборку аргументов команд под каждый эндпоинт (local vs `DOCKER_HOST=ssh://…`), (б) парсинг реальных примеров вывода `docker ps/images/compose ls --format json` (включая JSON-lines и пустой вывод). Live-тесты (`#[ignore]`) против локального Docker: список контейнеров, стрим логов, restart тестового контейнера.
- **`docker_ui`:** GPUI-тесты с фейком: дерево рендерится из фейковых данных; разрушающие действия заблокированы при `read_only`; на управляемом — требуется confirm перед вызовом; вью логов дописывает `LogChunk`; Error-состояние эндпоинта отображается.

## 11. Сборка и правила

- Крейты по конвенции репозитория: `[lib] path = "docker_client.rs"` / `docker_ui.rs`, без `mod.rs`.
- Локальные сборки форка требуют `--features gpui_platform/runtime_shaders` (на машине нет Metal/Xcode).
- clippy через `./script/clippy`.

## 12. Фазирование

1. **v1** — всё из §2 (UI, без MCP).
2. **Фаза 2** — `docker_mcp`: агент управляет докером через те же данные и тот же read-only/confirm-гейт.

## 13. Риски и открытые вопросы

- **Парсинг CLI-вывода** может дрейфовать между версиями Docker; смягчение — `--format json` (стабильнее табличного) и терпимый к неизвестным полям парсер.
- **SSH-латентность** на remote-эндпоинтах: поллинг реже, действия — по запросу; стрим логов — одно долгоживущее соединение.
- **`docker compose` vs `docker-compose`** (v2 плагин vs legacy бинарь): целимся в `docker compose` (v2); при отсутствии — понятная ошибка.
- Открытый вопрос на будущее (не блокирует v1): показывать ли git-ветку/commit образа из OCI-лейблов (`org.opencontainers.image.revision`) — отложено.
