# Self-Learning Zed Agent — Architecture Plan

## Overview

Two systems that turn Zed's native agent from a per-session tool into
a continuously improving, multi-model orchestration platform.

---

## 1. Skill Curator

### What it does

After every agent thread completes, a background task analyzes the
conversation and extracts reusable patterns into `SKILL.md` files.
Over time, the agent gets better at your specific project's conventions
without anyone writing rules by hand.

### Data flow

```
Thread completes
      │
      ▼
┌─────────────────────────────┐
│  Curator                      │
│                              │
│  1. Read thread messages     │
│  2. Extract:                 │
│     - Terminal commands run  │
│     - Files edited           │
│     - Commands used          │
│     - Errors encountered     │
│  3. Classify pattern type    │
│     (workflow / fix / config)│
│  4. Check for duplicates     │
│  5. Write SKILL.md           │
└──────────┬──────────────────┘
           │
           ▼
    ~/.agents/skills/<slug>/SKILL.md
           │
           ▼
    Loaded next thread (agent sees it)
```

### File layout

```
crates/agent/src/
├── curator/
│   ├── mod.rs              # Curator struct, entry point, config
│   ├── extractor.rs        # Parses thread messages for patterns
│   ├── classifier.rs       # Categorizes patterns (workflow/fix/config)
│   ├── dedup.rs            # Checks existing skills for overlap
│   ├── writer.rs           # Generates SKILL.md from extracted data
│   └── events.rs           # Hooks into thread lifecycle
```

### Key types

```rust
pub struct Curator {
    /// How often to run (default: after every thread completion)
    schedule: CuratorSchedule,
    /// Min pattern frequency before a skill is created (default: 2)
    min_occurrences: usize,
    /// Max skills before consolidation kicks in
    max_skills: usize,
}

pub enum PatternType {
    /// Repeating terminal command sequence (e.g. cargo test → cargo clippy)
    Workflow,
    /// Common fix for a recurring error
    Fix,
    /// Project-specific configuration (e.g. "uses pnpm, not npm")
    Convention,
}

pub struct ExtractedPattern {
    pub pattern_type: PatternType,
    pub commands: Vec<String>,
    pub files_touched: Vec<String>,
    pub frequency: usize,
    pub confidence: f64,
    pub summary: String,
}
```

### Heuristic examples

| Pattern detected | Generated skill |
|---|---|
| User runs `cargo test` → fixes compile errors → runs `cargo test` again 3+ times | `fix-rust-compile-errors: "When you see rust compile errors, first fix type mismatches, then lifetimes, then run cargo test to verify"`
| Agent always runs `pnpm lint --fix` before committing | `pre-commit-lint: "Before git commit, run pnpm lint --fix to auto-fix formatting"`
| Agent writes tests after editing Python files in `/src/` | `python-tests-after-edit: "After editing .py files in src/, run pytest to verify changes"`

### Thread hook integration

In `thread.rs`, when the agent loop finishes:

```rust
// At end of agent_loop(), before returning
if let Err(e) = Curator::new().observe(&self.messages, cx) {
    log::warn!("Curator observation failed: {e}");
}
```

This runs in a background task so it never blocks the user.

### SKILL.md output example

```markdown
---
name: fix-rust-compile-errors
description: >-
  When the compiler reports type or lifetime errors in Rust code,
  fix them systematically then verify with cargo test.
model_invocation: true
generated_by: curator
created_at: 2026-07-16
---

When you encounter Rust compilation errors:

1. Read the full compiler output — fix type errors first (they cascade)
2. Fix lifetime annotations next
3. Run `cargo check` after every 2-3 fixes
4. Once clean, run `cargo test`
5. If tests fail, fix and repeat from step 3

*This skill was auto-generated by the curator from repeated patterns*
```

## 2. Multi-Agent Router

### What it does

Inspects the user's prompt before the agent loop starts and routes
it to the most appropriate model. Edits → a strong coder. Research
→ a model with web search skills. Quick terminal ops → a cheap
fast model.

### Architecture

```
User prompt
      │
      ▼
┌─────────────────────────┐
│  Router                  │
│                          │
│  1. Classify task type   │
│     (edit / research /   │
│      terminal / general) │
│  2. Select model + tools │
│  3. Optionally spawn     │
│     sub-agent            │
└──────────┬──────────────┘
           │
     ┌─────┴─────┐
     │           │
     ▼           ▼
  Edit agent  Research agent
  (strong)    (search-enabled)
     │           │
     └─────┬─────┘
           ▼
     Results merged
```

### File layout

```
crates/agent/src/
├── router/
│   ├── mod.rs              # Router struct, classify(), dispatch()
│   ├── classifier.rs       # Prompt → TaskType (rules-based or cheap LLM)
│   ├── dispatch.rs         # Creates sub-agent with right model + tools
│   └── profiles.rs         # Model profile config (settings schema)
```

### Key types

```rust
pub enum TaskType {
    /// File editing, code generation, refactoring
    Edit,
    /// Web search, documentation lookup, research
    Research,
    /// Terminal commands, build, test, deploy
    Terminal,
    /// General Q&A, planning, explanation
    General,
}

pub struct ModelProfile {
    pub name: String,
    pub model_id: LanguageModelId,
    pub tools: Vec<String>,
    pub max_tokens: usize,
    pub temperature: f64,
}

pub struct RouterConfig {
    pub edit_profile: ModelProfile,
    pub research_profile: ModelProfile,
    pub terminal_profile: ModelProfile,
    pub general_profile: ModelProfile,
    pub enabled: bool,
}
```

### Classification approach

Phase 1 — rules-based (no token cost):

| Pattern in prompt | TaskType |
|---|---|
| Contains "edit", "change", "add", "refactor", "rewrite", file paths | `Edit` |
| Contains "search", "find", "research", "look up", "documentation" | `Research` |
| Contains "run", "build", "test", "deploy", shell commands | `Terminal` |
| None of the above | `General` |

Phase 2 — cheap classifier model (when rules are ambiguous):
Route to a small model (e.g. Gemini Flash / GPT-4o-mini) with one
prompt: `"Classify this request as one of: edit, research, terminal, general"`.

### Settings schema (`settings.json`)

```json
{
  "agent": {
    "router": {
      "enabled": true,
      "profiles": {
        "edit": {
          "model": { "provider": "anthropic", "model": "claude-sonnet-4" },
          "tools": ["all"]
        },
        "research": {
          "model": { "provider": "google", "model": "gemini-2.0-flash" },
          "tools": ["search_web", "fetch", "read_file", "grep"]
        },
        "terminal": {
          "model": { "provider": "openrouter", "model": "deepseek/deepseek-chat" },
          "tools": ["terminal", "read_file"]
        }
      }
    }
  }
}
```

### Integration with existing profiles

The router lives ABOVE the current Ask/Write/Minimal profiles. When
routing is enabled:

1. Router classifies the task
2. Selects the right `ModelProfile`
3. The `ModelProfile`'s `tools` list replaces the profile's tool allowlist
4. Agent runs with the selected model + tools

Users can disable routing and fall back to their current profile.

## 3. Implementation Order

### Phase 1 — Curator (days 1-5)

| Step | File | What |
|------|------|------|
| 1 | `curator/mod.rs` | Curator struct, config, `observe()` entry point |
| 2 | `curator/extractor.rs` | Thread message parser, pattern extraction |
| 3 | `curator/classifier.rs` | Categorize patterns (workflow/fix/convention) |
| 4 | `curator/dedup.rs` | Check existing skills before creating |
| 5 | `curator/writer.rs` | Generate SKILL.md, write to `~/.agents/skills/` |
| 6 | `thread.rs` | Hook `Curator::observe()` into agent loop exit |
| 7 | `curator/mod.rs` | Add config settings (min_occurrences, enabled) |

### Phase 2 — Router (days 6-12)

| Step | File | What |
|------|------|------|
| 1 | `router/mod.rs` | RouterConfig, Router struct, classify() + dispatch() |
| 2 | `router/classifier.rs` | Rules-based classification, optional cheap LLM fallback |
| 3 | `router/dispatch.rs` | Spawn sub-agent with selected model + tools |
| 4 | `agent_settings/` | Add router profile config to settings schema |
| 5 | `default.json` | Add default router profiles |
| 6 | `thread.rs` | Route at start of `agent_loop()` before model call |
| 7 | UI | Add router toggle + profile picker to agent settings page |

### Phase 3 — Polish (days 13-15)

| Step | What |
|------|------|
| 1 | Curator dedup: merge overlapping auto-skills |
| 2 | Curator: add `/curator` slash command (status, run, pause) |
| 3 | Router: add `/model` command to override per-thread |
| 4 | Memory + Curator integration: curator reads memory for context |
| 5 | Test suite for both systems |

## 4. Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Curator generates too many low-quality skills | Configurable `min_occurrences` threshold (default: 2). `/curator prune` command. |
| Router adds latency on every request | Rules-based classification is ~0ms. LLM fallback runs a tiny model. |
| Router tools mismatch (e.g. research agent gets no terminal) | Each `ModelProfile` has an explicit tool allowlist. Default profiles are conservative. |
| Memory + Curator double-writes | Curator is the only writer of auto-skills. Memory is facts; curator is patterns. Separate concerns. |
| Breaking prompt caching | Router only picks model+profile once at thread start. Cache is stable for the full thread. |
