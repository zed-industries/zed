# Mermaid Fallback Text Color Fix

## Problem

Our SVG renderer doesn't support `<foreignObject>`, so we strip them and use
fallback `<text>` elements generated by merman's
`foreign_object_label_fallback_svg_text`. These fallback elements are placed at
the **SVG root level**, outside the diagram's group structure. This means:

1. CSS selectors that depend on ancestry (`.section-0 text`, `.node text`, etc.)
   cannot reach the fallback text.
2. The fallback text retains `fill` attributes baked in by merman's internal
   black/white heuristic, which is often wrong for themed dark/light modes.

The CSS infrastructure is correct — `mindmap_section_css` in `style_transform.rs`
generates proper text colors using `text_color_for_background` from
`postprocess/util.rs`. The colors just can't reach the fallback text via CSS.

## What already works

- **Journey diagrams**: Fixed. The `strip_foreignobject` pass detects native
  `<text>` elements (merman renders these alongside foreignObjects for journeys)
  and strips the redundant fallback groups. The native text gets correct colors
  via CSS (e.g., `text.journey-section, text.task { fill: {text} }` in
  `style_transform.rs`).

- **Flowcharts**: Node shapes use theme colors from the mermaid config
  (`primaryColor`, etc.). Text inherits from the SVG root `fill` which is set to
  `theme.text_color`. This works because flowchart nodes don't use per-node
  accent colors — they all share the same primary background.

## What needs fixing

### Mindmap text colors

Mindmap nodes are grouped under `.section-0`, `.section-1`, etc. Each section
has a distinct accent fill from `git_branch_colors`. The correct text color for
each section is already computed in `mindmap_section_css` using
`text_color_for_background`, but the CSS rule `.section-{i} text` doesn't match
the fallback `<text>` at the SVG root.

The fallback text has `fill="black"` or `fill="#ffffff"` from merman — a crude
heuristic that's wrong for many theme/color combinations.

**Fix approach**: A postprocessing pass that matches each fallback `<text>` to
its parent mindmap section (by position — see the prototype in the commented-out
`accent_colors.rs`) and sets the fill to the value from `text_color_for_background`.

### Class diagram text colors

Same structural issue. Class diagram nodes have per-node accent colors, and
fallback text for class names, members, and methods sits outside the node groups.

**Fix approach**: Same position-matching strategy. The commented-out
`accent_colors.rs` has a working prototype using `NodeRect` bounding boxes
parsed from the node's `transform` and first `<path>` element. Note: class
diagram text is offset from the node center (title, members, methods are at
different y-offsets), so exact position matching doesn't work — bounding box
containment is needed.

### Sequence diagram text colors

Sequence diagram actor labels use `<text><tspan>` where the `<tspan>` gets a
fill from merman's CSS that overrides the accent color set on the parent
`<text>`. The commented-out `accent_colors.rs` handles this by propagating the
accent fill to `<tspan>` elements inside accented `<text>` elements
(`current_text_accent` field).

### ER diagram text colors

Not yet investigated in detail, but likely has the same fallback text issue as
class diagrams.

## Implementation plan

### Step 1: Create per-diagram submodules

Replace the monolithic `postprocess/accent_colors.rs` with a submodule per
diagram type under a new `postprocess/accent_colors/` directory:

- `postprocess/accent_colors/mindmap.rs`
- `postprocess/accent_colors/class_diagram.rs`
- `postprocess/accent_colors/sequence_diagram.rs`
- `postprocess/accent_colors/er_diagram.rs` (if needed after investigation)

Each submodule should:

1. Detect its diagram type from the SVG root's `class` attribute
2. Only activate for its specific diagram type
3. Match fallback `<text>` elements to their parent nodes/sections
4. Set the correct `fill` using `text_color_for_background` from `postprocess/util.rs`

A thin orchestrator in `postprocess/accent_colors.rs` (re-enabled) should chain
these per-diagram passes, or run the appropriate one based on the detected
diagram type.

### Step 2: Implement each diagram type in parallel

Since each submodule has a disjoint write scope (its own file) and operates on
different diagram types, all implementations can be done in parallel — one
subagent per diagram type. Each subagent should:

1. Read the existing prototype code in the commented-out `accent_colors.rs` for
   patterns (position matching, `NodeRect` bounding boxes, `current_text_accent`
   for tspan propagation, etc.)
2. Implement the pass for its diagram type
3. Un-ignore and update the relevant test(s) in `tests/debug_accent_flowchart.rs`
4. Verify with `cargo test -p mermaid_render` and `cargo clippy`

## Architecture notes

### The commented-out accent_colors pass

`postprocess/accent_colors.rs` is a complete (but commented-out) implementation
that handles most of the above. It was disabled because it applied accent colors
to ALL diagram types including flowcharts and ER diagrams, where the result
looked wrong. The key insight for re-enabling it:

**It should be opt-in per diagram type.** The SVG root element has a `class`
attribute that identifies the diagram type (`"flowchart"`, `"classDiagram"`,
`"statediagram"`, `"mindmap"`, etc.). The pass already has an `accent_nodes`
flag that disables node coloring for flowcharts — this pattern should be extended
so each diagram type gets the appropriate behavior.

### The `text_color_for_background` utility

`postprocess/util.rs` exports `text_color_for_background(Hsla) -> Hsla`. It uses
OKLCH color space to produce a text color that is:

- Subtly tinted by the background's hue (not flat black/white)
- Guaranteed ≥ 4.5:1 WCAG AA contrast ratio (verified by a property test)
- Falls back to pure black/white only for edge-case backgrounds near the
  luminance boundary

This is re-exported as `mermaid_render::text_color_for_background` and used by
`mindmap_section_css`, `git_branch_label_colors`, and `build_mermaid_theme`.

### Key files

| File                                 | Role                                                                       |
| ------------------------------------ | -------------------------------------------------------------------------- |
| `postprocess/accent_colors.rs`       | Commented-out pass with position-matching prototype                        |
| `postprocess/util.rs`                | `text_color_for_background` OKLCH algorithm + property test                |
| `postprocess/style_transform.rs`     | `mindmap_section_css` and `build_injected_css`                             |
| `postprocess/strip_foreignobject.rs` | Strips foreignObjects; also strips fallback groups when native text exists |
| `postprocess/fallback_fixup.rs`      | Processes fallback groups (entity fixes, rect recoloring)                  |
| `postprocess.rs`                     | Pipeline orchestration                                                     |
| `render.rs`                          | Merman config with `cScale`/`cScaleLabel` and theme variable mapping       |
| `markdown/src/mermaid.rs`            | `build_mermaid_theme` — constructs `MermaidTheme` from Zed's UI theme      |

### Diagram type detection

The SVG `<svg>` element's `class` attribute identifies the diagram type. Known
values from merman:

| Class          | Diagram type                         |
| -------------- | ------------------------------------ |
| `flowchart`    | Flowchart / graph                    |
| `classDiagram` | Class diagram                        |
| `statediagram` | State diagram                        |
| `mindmap`      | Mindmap                              |
| `journey`      | User journey                         |
| (empty)        | Sequence diagram, ER diagram, others |

### Tests

- `tests/debug_accent_flowchart.rs` has two `#[ignore]`d tests
  (`class_diagram_fallback_text_uses_accent_colors` and
  `sequence_diagram_tspan_uses_accent_text_color`) that assert correct accent
  text colors. Un-ignore these when re-enabling the accent pass.
- `postprocess/util.rs` has a `#[gpui::property_test]` that verifies WCAG AA
  contrast for all opaque backgrounds.
